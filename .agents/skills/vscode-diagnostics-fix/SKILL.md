# VS Code Diagnostics Fix

## Workflow

1. Identify the target file.
   - "open file", "active file", or "current file" means the IDE active file.
   - If the user names a path, use that path instead.
2. Call VS Code diagnostics MCP `get_errors` and `get_warnings`.
3. Filter diagnostics to the target file by normalized absolute path.
   - If there are no error or warning diagnostics for the target file, say so and stop.
   - Do not fix unrelated files just because they appear in workspace diagnostics.
4. Read the full target file from disk and treat it as the source of truth.
   - Include staged and unstaged edits.
   - Preserve unrelated local work.
5. Fix only the reported error and warning diagnostics for the target file.
   - Do not rewrite working code just to satisfy style preferences.
   - Do not make broad refactors unless the diagnostic cannot be fixed safely otherwise.
6. Rerun VS Code diagnostics MCP `get_errors` and `get_warnings`.
7. Repeat only for remaining error or warning diagnostics on the target file.
8. Summarize:
   - The original target-file diagnostics.
   - The files changed.
   - The exact VS Code diagnostics MCP checks run.
   - The final target-file diagnostics result.

## Guardrails

- Fix errors and warnings only. Do not inspect or fix information, hints, or suggestions unless the user explicitly asks.
- Do not treat informational diagnostics as required fixes.
- Do not use stale terminal output as the source of truth when VS Code diagnostics MCP is available.
- If diagnostics remain because they are generated by an unavailable extension, stale language-server state, or external dependency, say that directly and include the exact diagnostic text.
- Do not run lint, test, build, or separate validation commands unless the user explicitly asks for them in the same request.
